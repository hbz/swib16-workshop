Software development in libraries

Single person, internal teams, cross-organization projects

Factually always many people are involved in development

How do we get together?

Transparent & open

Transparent: Let others see what you're doing; make source code available, have public issue tracking and discussions etc.

Open: Let others actually participate; this is the harder part: encourage participation, give up absolute power

Being transparent is easier, and it's the foundation

GitHub

Something like a social network, to
"learn, share, and work together to build software"

Lets you share your own projects

Lets you collaborate on projects you or other people have shared

GitHub helps a lot with being transparent and open

GitHub Flow

Used by GitHub to develop GitHub

Git makes branching and merging easy

Implement new features and bugfixes in separate branches

 
 
 

Exercise X-b:
Propose your fix

  1. Create a new local branch

    
git checkout -b bugfix-issue-1

  2. Commit your changes to the branch

    3.

  3. Push your branch to the remote

    
git push origin bugfix-issue-1

  4. Open a pull request for the branch

    5.

Our review process

Functional review: review actual functionality, associated with the issue, done by librarian

Code review: review code changes, associated with the pull request, done by developer

https://hbz.github.io/#dev-process

Exercise X-c:
Review your fix

Assign issue to librarian, with instructions on how to review; review

Assign pull request to developer; review

Both positive: merge the pull request
